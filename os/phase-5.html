Key Decisions — Locked
Architecture: hybrid, same backend, separate surface
Rejected fully-standalone app (duplicates OAuth/hosting) and a folded-in dashboard tab (couples experimental feature to stable ops tool). New route/panel in the same northlight-dispatcher deployment, reusing existing Drive/Gmail/Calendar/ClickUp auth.
Model: Haiku + Sonnet mix, not Haiku-only or external model
Real cost math showed Sonnet is negligible at this volume (~$0.10–0.15/session) — quality matters more than the marginal cost difference. External/non-Anthropic model rejected: pennies of savings, new vendor relationship, against Elizabeth's own stated discomfort with adding tools outside the existing relationship.
$5/month hard usage cap
Goal is as-close-to-$0-as-possible; $5 is the ceiling, enforced in code (session-state.js) before every session and round — not just a documented intention.
Phase 1 board: CFO, Privacy Officer, CMO, Secretary
CMO pulled into Phase 1 against the original recommendation to defer it, because marketing decisions were Elizabeth's stated priority. CFO/Privacy/Secretary needed no new data integrations; CMO reuses existing Plausible/LinkedIn/Threads pulls already wired for the OS.
Knowledge Base gate enforced as a hard blocker, not a soft note
Implementation was paused entirely mid-session when ratified mission/values/principles didn't exist. Resumed only after Elizabeth ran a real chat interview and produced ratified docs — confirms personas need to represent real, sourced principles, not inferred ones.
Storage: Drive-only for Phase 1, GitHub explicitly deferred
No GitHub push capability exists in the codebase yet. Building it is real, separate work tied to the parallel-track Knowledge Base project — not bundled into Advisory Board Phase 1.
Self-check: one pass, with an actual regeneration retry — not multiple iterative rounds
Elizabeth confirmed one pass for v1, with room to expand to multiple rounds later for specific sessions. Built as a configurable parameter (refinementPasses), not hardcoded.
Persona assignments are single-voice, not multi-voice
An assignment is a deliverable from one specific persona, not a debate — voicing multiple personas would blur the distinction the orchestrator already protects in regular rounds.
"Reference past decisions" shipped as a cheap context-injection MVP now, not deferred entirely to Phase 3
Full relatedDecisions resolution is real future work, but listing the 8 most recent decisions in every new session's context answers Elizabeth's actual question today at near-zero added cost.
Single-shot assignments replaced entirely with multi-turn threads (not added alongside)
Elizabeth's actual need ("meet 1:1 and debate the answer") was a different shape than the original single-shot design, not an enhancement to it — kept one clean implementation rather than maintaining two assignment patterns.
Thread persistence upserts the same Drive file, not a new file per reply
A 5-message back-and-forth shouldn't produce 5 separate Drive files — one thread, one file, updated in place. saveAssignmentRecord takes an optional fileId for this.
